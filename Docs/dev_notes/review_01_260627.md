# fusion-ocr — Adversarial Code Review

_Date: 2026-06-27_

Scope: architecture, every source module, the test suite (run: 42 passed, 9 skipped),
the docs, and the dependency/model choices.

## Overall verdict

The design instinct is genuinely good — deterministic geometry as an anti-hallucination
gate, Document-in/Document-out stages, provenance retention, explainable routing. The
skeleton is clean and the tests are fast and dependency-free. But the project's
*credibility claims* (trustworthy, defensible, reproducible, "re-run with a refined
prompt") outrun what the code actually delivers, and several of the load-bearing
decisions don't survive contact with the workflow they're meant to serve.

Findings are ordered from highest leverage down.

---

## 1. The resume/idempotency model defeats the headline workflow

This is the most important finding because it undermines the stated purpose.

`pipeline.py` (`process()`, ~L73–86) resumes "from the last completed stage," and work is
keyed solely on the PDF's content hash. The docstring and README sell this as: *"a
deliberate re-run with a refined prompt resumes from the last completed stage instead of
redoing OCR."*

It does the opposite. If `stage_completed == "render"` (a finished doc), re-running skips
**every** stage, including `vlm_read`. There is no mechanism to invalidate *from* a chosen
stage, so the advertised "refine the VLM prompt and re-run, reusing the OCR" is
unachievable without hand-editing `doc.json`. Worse, via the watcher/API the job never
even reaches `process()`: `watcher.scan_once` (~L26–28) and `jobs.upsert_queued` (~L39–54)
short-circuit on the existing hash. So re-dropping the same bytes is a total no-op
**regardless of config, prompt, model, or code changes**.

For an iterative investigative tool this is backwards. The cache key is "same input bytes"
when it needs to be "same input *and* same processing recipe" (model + prompt + routing +
code version). As written, the only way to reprocess after tuning a prompt is to manually
delete `jobs.sqlite` rows and `out/<sha>/doc.json`. That contradicts the defensibility
principle that re-analysis with refined prompts should be cheap.

Secondary fragility in the same loop: resume is matched by stage *name* (`seen_completed`,
~L74–80). If a stage is renamed/removed/reordered between runs, `seen_completed` never
flips true and **all stages are silently skipped**, returning a stale doc as if freshly
done.

**Fix direction:** make the cache key a fingerprint of (content hash + pipeline version +
per-stage config/prompt hash); add an explicit "rerun from stage X" / `--force` path;
validate that `stage_completed` exists in the current pipeline.

## 2. The API is not safe to expose, and it advertises that it will be

`api.py` (`submit`, ~L33–48) is described as "the stable contract consumers (Giant) call …
moving it [to a VPC] is invisible to callers." Three problems:

- **Path traversal.** `dest = in_dir / (pdf.filename or "upload.pdf")` (L34) uses the
  attacker-controlled multipart filename unsanitized. `filename="../../something"` writes
  outside `in/`. Use `os.path.basename` (or a generated name).
- **Blocks the event loop.** `async def submit` calls the synchronous, multi-minute
  `process()` directly. With uvicorn's default single worker the whole server stalls per
  request — no concurrency, no health checks during a VLM read. Offload to a real queue or
  `run_in_executor`.
- **Unbounded ingest + no auth.** `await pdf.read()` loads the entire upload into memory
  (trivial DoS), there's no size/content-type check, and no authentication. Fine for
  loopback today; the moment it's "invisibly" moved to a VPC it's an open, blockable
  endpoint.

The `GET /jobs/{sha256}` handler also feeds an unvalidated path component into
`Path(out_dir)/sha256` + `iterdir()`.

## 3. No ground-truth evaluation; the accuracy story is anecdotal

For a tool whose entire pitch is *trustworthy, defensible* OCR, there is no held-out eval
set with ground truth. Every claim in the docs is a single hand-checked document:
"Verified on the Thai scan," "proven on the handwritten note (6 → 3,185 searchable
chars)." That last figure measures *quantity recovered*, not correctness — it says nothing
about character error rate or hallucination rate. The roadmap lists "the 'Giant rejects'
eval (old vs new)" as still to-do, which confirms the gap.

Compounding this:

- **Uncalibrated magic thresholds everywhere.** `_PUA_MAX=0.01`, `_LARGE_IMAGE_FRAC=0.40`,
  `_BODY_COVERAGE_MIN=0.50`, `_MR_COVERAGE=0.5`, `_IOU_OVERLAP=0.5`,
  `_NONLATIN_MIN_SHARE=0.10`, cluster band `0.6*h`, `apple_vision_skip_vlm=0.92`, refusal
  `len(r) < 0.25*det_chars`, `_CONTAINED=0.7`, `_TOUCHES=0.1`. Triage, routing, fusion, and
  table calibration all hinge on hand-picked constants that appear tuned (implicitly) to a
  handful of sample docs and are mostly not config-exposed. This is the core versatility
  risk: behaviour on a new document class is unpredictable.
- **Nothing is pinned.** `pyproject.toml` pins no versions (`paddleocr`, `paddlepaddle`,
  `httpx`, `fastapi`, …). PaddleOCR in particular has had breaking 2.x→3.x churn — which
  the code already works around in `ocr_det.py` (~L48–66). For a provenance tool, an
  unpinned dependency stack means the same PDF can yield different OCR after a silent
  upgrade. That directly contradicts the defensibility goal — outputs aren't reproducible
  across time or machines. Model identifiers (Qwen3-VL-8B-**4bit**, Typhoon) are at least
  named, but the 4-bit quant of the *default reader* trades accuracy for speed with no
  measured error delta.

If defensibility is the selling point, "reproducible + measured" has to be a first-class
property, not a roadmap bullet.

## 4. The anti-hallucination guarantee is narrower than the framing

The README says the deterministic layer is "the anti-hallucination gate" and "keeps the
VLM from inventing text where there was no ink." True for the *overlay/segments* — VLM
lines that don't align to a detected box are dropped. But:

- **The markdown reading view is ungated.** `render.py` (~L139–141) emits
  `page.vlm_reading` verbatim for OCR pages. The clean reading a journalist actually reads
  is the raw VLM transcription, hallucinations included. The gate protects the artifact
  nobody reads and not the one they do.
- **Unaligned clusters inject raw OCR garbage into the "trustworthy" overlay.** In
  `fusion.py` (~L156–159), any fused cluster with no aligned VLM line falls through to
  `seg.best_text = seg.det_text`. For a handwriting/degraded page where Paddle's `det_text`
  is noise, that noise becomes searchable overlay text. So the overlay silently mixes
  VLM-clean lines with un-corrected OCR; markdown and overlay can disagree. (Provenance
  survives — `source` stays `paddle`/`vision` not `fused` — so a careful consumer *can*
  tell, but nothing surfaces it.)
- **Alignment is weakest exactly when it matters.** The whole join is Needleman–Wunsch over
  fuzzy similarity of the VLM line vs the box's `det_text` (`_nw_align`, ~L120–145). When
  `det_text` is garbage — the precise case you brought the VLM in for — the signal driving
  placement is weak, so the correspondence between a searched word and its highlighted box
  is shaky. The Thai↔Thai recogniser trick is a smart mitigation, but it doesn't exist for
  handwriting or unsupported scripts. Worth stating plainly: you get reliable
  *existence/extent* of ink, but *which box a given word lands on* is soft.

Minor DP smells in the same function: floating-point exact-equality in the traceback
(~L138) and `_sim` recomputed on every diagonal in both the fill and the traceback
(doubles the SequenceMatcher cost, which is the hot path on dense pages).

## 5. The airgap guard has real holes for the "most-sensitive tier"

`config.enforce_airgap` (~L93–117) monkeypatches `socket.socket.connect`. Clever, but for
the tier whose whole point is zero egress:

- **DNS still leaks.** Only `connect` is patched, not name resolution. If `vlm.base_url`
  (or escalation/probe endpoints) uses a hostname like `transcription-gpu.internal`,
  `getaddrinfo` egresses the query *before* connect refuses it. The configured resolver
  learns what you're querying.
- **`connect_ex` is unpatched**, so code paths using it bypass the guard entirely.
- **AF_UNIX breakage.** For a Unix-domain socket, `address` is a path string, so
  `str(host) not in _LOOPBACK` (L109) is true → the guard raises `OSError` on legitimate
  local IPC.
- **Loopback set is too literal.** `{"127.0.0.1","::1","localhost"}` rejects the rest of
  `127/8` and `::ffff:127.0.0.1`.
- **Failures degrade silently.** A refused connection surfaces as `except Exception:
  return ""` in `vlm_read` (~L94–98), so "airgap blocked the reader" looks identical to
  "the page had no text" — the page just quietly falls back to `det_text` with no signal
  that the sensitive-tier reader was unreachable.

Also: patching a stdlib method process-wide is irreversible within the run, and the test
that triggers it (`test_pipeline.py`, ~L48–53) leaves `socket.connect` patched for the
remainder of the pytest process — latent test-order pollution.

## 6. The serialize-between-stages contract forces 4–5× redundant rasterization

Because the resume contract is Document→JSON between stages, rendered page images can't be
passed in memory, so every stage that needs pixels re-opens the PDF and re-rasterizes from
scratch, at *different* DPIs: layout 150, table 150, language probe 120, ocr_det 200,
vlm_read 150. A single image-only page is rasterized 4–5 times (each a fresh `fitz.open`).
Rasterization is among the most expensive steps; this is a direct, unacknowledged cost paid
for the resume feature. A shared page-image cache (keyed by page+DPI, scoped to one run)
would remove most of it without touching the resume model.

Related: `doc.json` is rewritten after every one of the 9 stages (`pipeline.py`, ~L81–84).
For a 100-page scan with thousands of segments that's a multi-MB `asdict` + JSON dump 9×
per run.

## 7. The watcher isn't fit for the drop-folder workflow it implements

`watcher.scan_once` (~L23–40):

- **Re-hashes the entire `in/` folder every tick** (default 2s). Dedup happens *after*
  hashing, and processed files are **never moved out of `in/`**, so every file is re-hashed
  forever. O(files × bytes) every 2 seconds — a real scaling wall.
- **No file-stability check.** A PDF still being copied in can be hashed mid-write and
  processed as a truncated "job." Classic drop-folder race; needs a size/mtime-stable-for-N-
  ticks gate or an atomic move-on-complete.

## 8. Concurrency claims vs concurrency safety

The docs say the router/API "scales to a real queue later" and the job table "scales to a
real queue." But `jobs.upsert_queued` (~L39–54) does SELECT-then-INSERT with no
transaction/atomicity. Two uvicorn workers (or watcher + API) on the same file race: both
see no row, both INSERT, one raises `sqlite3.IntegrityError` (unhandled → 500). Use
`INSERT … ON CONFLICT DO NOTHING`. SQLite is also opened without WAL, and two processes
would clobber the same `out/<sha>/doc.json` and pixmaps. The current code is
single-process-only; the "scales later" framing hides that even two workers aren't safe
today.

## 9. Serialization, client, and rendering smells

- **Hand-rolled `from_json`** (`models.py`, ~L97–137) mirrors `asdict` field-by-field. Add
  a field to any dataclass and forget to update `from_json` → it's silently dropped on
  every resume. Dual-maintenance with a silent failure mode; a schema-driven (de)serializer
  would remove the footgun.
- **VLM client** (`vlm/openai_compat.py`, ~L23–48): no `max_tokens` (a pathological page
  can generate until the 600s timeout — latency/cost risk on a shared GPU), no
  retry/backoff (a transient 503 becomes a silent empty read → `det_text` fallback), a
  fresh `httpx.Client` per page (no keep-alive), and PNG-only base64 (a 150-DPI full page
  is multi-MB, +33% for base64 — painful for the "VPC later" path; JPEG would be far
  smaller for scans).
- **Overlay word-granularity is positionally wrong and oversold.** `overlay/pymupdf_overlay.py`
  (~L106–113) spaces words by equal `span/len(words)` regardless of glyph width (the
  docstring calls this "precise highlights" — it isn't), and it always advances along `x`
  (`x0 + i*step`) even on 90/270° pages where it should advance along `y`. Latent since the
  default is `line`.

## 10. Doc/code drift and partial provenance

- `Docs/routing.md` (L4) and `routing.py` (L4) both assert *"PaddleOCR is ALWAYS the
  geometry engine."* That's now false — `prefer_apple_vision` makes Apple Vision the
  geometry engine via `_auto_engine` (~L101–108). The invariant you actually hold is
  "geometry is always deterministic," which is the better claim; the literal "always
  PaddleOCR" wording is stale.
- **Provenance is only partly realized.** routing.md's principle 2 says every passage backs
  to *the reason its reader was chosen*. The index records page `script`, `read_model`, and
  per-segment `read_by`, but not the resolved `paddle_lang`/`engine` or *why* — so the
  routing decision isn't fully auditable from the artifact.
- The `Box` quad "tracks skew/rotation" (`models.py`, ~L23–35), but fusion, compose, and
  overlay collapse everything to axis-aligned `bbox`. The skew is stored and then discarded
  downstream, so the docstring oversells what's used.

## 11. Triage heuristic edge cases

`triage._max_image_frac` (~L107–116) measures the *largest single* image, not total image
coverage. A scan tiled into strips (common from some producers) can have every strip < 40%
while the page is fully a scan → `large_image=False` → a partial text layer suppresses OCR
of the body. Conversely a full-page decorative background on a born-digital page can force
needless OCR. The region machine-readable test in `compose.region_text_coverage` (~L40–49)
is similarly brittle: real paragraphs rarely fill ≥50% of their bbox once you account for
leading and margins, so a genuinely machine-readable region can score under threshold and
be mis-tagged `ocr`.

## What's genuinely good (for calibration, not flattery)

- The `Stage` protocol with clean dependency-free passthroughs is a real strength — the
  skeleton runs and is testable without any model, and the tests exercise the hard logic
  (alignment, mixed-content composition, XY-cut, confidence calibration) rather than just
  happy paths.
- Provenance retention (`superseded`, `det/vlm/best`, `read_by`) and the "never discard
  source" discipline line up well with the investigative-pipeline principles.
- Deterministic, explainable routing and XY-cut reading order (over a black-box model) is
  the right editorial call.
- The MLX-vs-Ollama empiricism and the Thai PaddleOCR-th + Typhoon route are solid,
  concrete engineering.
- Apple Vision as a cheap on-device tier with a confidence gate is a smart cost/latency
  optimization.

---

## Prioritized recommendations

1. **Re-key the cache** on (content + pipeline version + stage config/prompt hashes); add
   explicit re-run-from-stage / `--force`. Without this the iterate-on-prompt workflow
   doesn't function.
2. **Fix the API**: sanitize the upload filename, offload `process()` off the event loop,
   add size limits + auth before any non-loopback deployment.
3. **Stand up a real eval**: a small labeled corpus with CER/WER + a hallucination check,
   and **pin** dependencies and model versions. This is what makes "defensible" true rather
   than aspirational.
4. **Gate the markdown** the same way as the overlay (or clearly mark ungated VLM prose),
   and stop letting unaligned clusters silently emit raw `det_text` into the overlay.
5. **Close the airgap holes** (block resolution, `connect_ex`, AF_UNIX, full loopback
   range) and make a blocked sensitive-tier read a loud error, not a silent `det_text`
   fallback.
6. **Cache page rasters within a run** and stop rewriting `doc.json` after every stage for
   large docs.
7. **Harden the watcher**: track seen files, require file stability, move processed files
   out of `in/`.
