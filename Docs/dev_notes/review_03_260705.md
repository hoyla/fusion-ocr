# Adversarial Project Review 03

_Date: 2026-07-05_

Scope: the whole project, strategy included — not just the code. Method: full read of
src/ and tests/, all docs (treated as claims to verify, not facts), an audit of the
evidence base behind every empirical claim, and a web survey of the mid-2026
document-AI ecosystem (competitors, engines, table/reading-order tooling). Read-only —
no changes were made. Questions posed by the owner:

1. Is the approach appropriate for the constraints (offline-capable; Mac or Mac
   cluster) and requirements (hybrid deterministic+VLM OCR API across document types)?
2. Are the engines the right choices?
3. Have we orchestrated well-researched third-party tools, or are we reinventing
   wheels on limited knowledge and a limited corpus?
4. Can performance improve, and at what quality cost? Is a Python-focused pipeline
   right?
5. Flaws, omissions, oversights in construction and code?
6. Is this a waste of time — has someone already built it?

## Verdict

The architecture is right, genuinely differentiated, and (checked adversarially, not
assumed) not duplicated anywhere as of mid-2026. The engineering discipline —
provenance model, recipe-fingerprinted resume, traceable remediation of review
findings — is real. The project has two serious weaknesses, neither the one usually
feared:

1. **The empirical base has not kept up with the claims built on it.** The entire
   measured evidence is under 40 scored items (heavily one FOI document family, almost
   all English), while ~1,200 usable gold items sit on disk unused — and the two
   promises the architecture exists to deliver (a searched word highlights the *right
   box*; the reading view's hallucination rate is bounded) have **no metric at all**.
2. **Wheel-reinvention has happened at the implementation layer, not the concept
   layer.** The bespoke concepts are correctly bespoke; their internals hand-roll
   things `rapidfuzz`, `jiwer`, and OCRmyPDF's renderer do better, studded with ~15
   uncalibrated constants tuned on a handful of documents.

Both are fixable without touching the architecture.

---

## 1. Approach vs constraints and requirements

**The approach — yes. The implementation covers less of the requirement space than the
docs imply.**

The constraint logic in motivation_and_strategy.md holds: confidentiality rules out
mature cloud OCR; deterministic geometry + VLM reading + inference-time gating is a
sound answer to "VLMs read what nothing else can but can't be trusted alone." The
ecosystem survey *strengthened* this: MinerU added a "hybrid" backend in 2025–26
explicitly citing lower hallucination risk; Unstructured shipped element-level VLM
routing for the same reason; olmOCR's changelog fixed blank-page hallucination. The
failure mode we designed against is one the whole field now concedes.

Gaps against the stated constraints/requirements:

- **Airgap: the seal is Python-deep only.** `config.enforce_airgap` patches
  `connect`/`connect_ex`/`getaddrinfo` — but not `gethostbyname`/`gethostbyname_ex`,
  not UDP `sendto`, and nothing at the C level: paddlepaddle, onnxruntime, or any
  native dependency can call `connect(2)` directly; subprocesses are unguarded. For
  the most-sensitive tier this is defence-in-depth theatre unless paired with an
  OS-level control (pf rules, a network-less user, `sandbox-exec`). Docs currently
  oversell what the seal guarantees.
- **Mac cluster: effectively no story.** The seams (JobStore, storage.py, VLM
  base_url) are honestly shaped, but the only conceivable multi-machine mode today —
  shared folder + `jobs.sqlite` — is actively dangerous: **SQLite WAL does not work on
  network filesystems (SMB/NFS)**. The only component that can live on another Mac
  today is the VLM endpoint. Fine under *look before infra*, but "cluster of Macs if
  necessary" is aspiration, not capability, and nothing guards the broken interim.
- **Requirement coverage:** Office docs not ingestible (roadmap item); encrypted /
  password PDFs unhandled (fail confusingly deep in a stage); **HEIC** — what an
  iPhone photo of a document actually is — not sniffed (nor WebP); triage's OCR
  decision is **page-level**, so a small embedded scan (<40% of a text page, or layer
  coverage ≥50%) is silently never OCR'd — uncomfortably close to the mixed-content
  failure that motivated the project (triage.py ~L100).
- **The "API" has no artifact-retrieval endpoint.** `GET /jobs/{sha}` returns artifact
  *names* only (api.py ~L123, storage.py ~L19); a remote consumer (Giant) cannot fetch
  `document.md`/`overlay.pdf` over the "stable contract" — a shared filesystem is
  assumed. `storage.artifacts` also lists internal resume snapshots as "artifacts".

## 2. Engine choices

**Broadly right — the survey validates the picks — but one generation behind on two of
them, which is exactly the trap principles.md warns about ("audit the defaults,
periodically").**

- **PaddleOCR as deterministic baseline:** right, independently corroborated (MinerU's
  pipeline backend moved to PP-OCRv6; Docling supports Paddle-family engines). But
  **PP-OCRv6 shipped ~June 2026**: vendor numbers claim +5% recognition and a **6.1×
  speedup on Apple M4 for the tiny tier**. If that survives our eval, it may deliver
  most of what the RapidOCR migration is chasing *inside the engine we already trust*,
  with no port risk. Benchmark PP-OCRv6 tiny/small in the same A/B harness before (or
  alongside) RapidOCR.
- **RapidOCR:** still worth the A/B, with adjusted expectations. CoreML-EP
  practitioner reports (2026) say dynamic-shape graphs — which OCR detection
  inherently has — fall back to CPU or force subgraph splitting, so the realistic win
  is leaner ONNXRuntime-CPU, not the ANE. Measure CPU EP first; treat CoreML EP as a
  bonus experiment. The layout/order-head port risk is confirmed real:
  **RapidAI/RapidDoc** ships ONNX conversions of PP-DocLayoutV3/V2 + table models, but
  it is young (~195 stars) and whether the learned order head survives conversion
  needs the verification the eval plan already demands.
- **Layout: PP-DocLayoutV3 supersedes our V2** — instance segmentation (polygons —
  curved/tilted scans) with detection + reading order *jointly trained*, vs V2's
  decoupled pointer network. Same "coasted on an old default" failure we already
  caught once with PP-DocLayout_plus-L.
- **VLM:** Qwen-family via MLX matches 2026 community consensus; the MLX-vs-Ollama
  finding matches independent reports. Two caveats: the default switch to Qwen3.5-9B
  was made on Δ=0.005 aggregate recall over **4 pages** — inside any plausible noise
  band (mitigated by the one-line rollback, but the decision standard matters); and
  **the 4-bit quantisation has never been A/B'd against 8-bit** (review_01 flagged
  this; dropped). Every accuracy number in the repo is the quantised build's.
- **Worth testing as new tiers:** **PaddleOCR-VL-0.9B** now has an official
  Apple-Silicon path via mlx-vlm — a cheap structured reader between Apple Vision and
  the 9B generalist. Apple's WWDC25 `RecognizeDocumentsRequest` adds paragraph /
  table / list structure to Vision — a potentially free upgrade for the fast tier we
  use as flat det/rec.

## 3. Orchestration vs wheel-reinvention

**Concept level: succeeded, better than most.** Layout / table / OCR are thin wrappers
over public Paddle APIs (table.py's "no reimplemented layout logic" checks out); the
hand-rolled XY-cut was deleted in favour of the model's order head; the genuinely
bespoke pieces (ink-gate, supersede/compose, routing seam, fingerprinted resume,
provenance model) pass the principles.md build-only-if test — the ecosystem survey
confirms nothing off-the-shelf does them.

**Implementation level: four violations.** Bespoke *concepts* implemented with
hand-rolled *internals* that mature libraries do faster and more correctly:

1. **fusion.py hand-rolls Needleman-Wunsch twice** (`_nw_align` ~L129,
   `_word_distribute` ~L159), scored by a fresh `difflib.SequenceMatcher` per DP cell,
   with the backtrace *recomputing* similarity per diagonal. `rapidfuzz` provides
   C-speed `ratio` and editops. The `_MAX_DP_CELLS` guard exists only because the
   implementation is slow — and its fallback is a *worse* alignment, so slowness is
   paid for in quality.
2. **eval/metrics.py hand-rolls Levenshtein + backtrace** where `jiwer` /
   `rapidfuzz.distance.Levenshtein.editops` are standard. (Correct, verified — but a
   pure-Python wheel.)
3. **The overlay is a self-acknowledged partial OCRmyPDF reimplementation**, and its
   `granularity="word"` mode places words at equal-width steps regardless of word
   length (pymupdf_overlay.py ~L107) — wrong geometry that OCRmyPDF's hOCR renderer
   gets right. Latent while the default is `line`.
4. **`detect_script` hand-rolls Unicode ranges and misses Arabic presentation forms**
   (U+FB50–FDFF, U+FE70–FEFF — very common in real PDF text layers), CJK extensions,
   fullwidth forms (routing.py ~L44). The `regex` module's script properties / PyICU
   do this exhaustively.

**The deeper form of the owner's worry is confirmed.** ~15 magic constants —
`fuse_min_sim=0.34`, `fuse_det_conf_trust=0.80`, `apple_vision_skip_vlm=0.92`,
`_PUA_MAX=0.01`, `_LARGE_IMAGE_FRAC=0.40`, `_MR_COVERAGE=0.5`, `_ANCHOR_SIM=0.5`,
`_MIN_ANCHOR_FRAC=0.3`, the `0.6×h` line-band constant tuned independently in
fusion.py ~L109 *and* eval/datasets.py ~L79, … — have **zero calibration evidence**;
none are in `recipe_fingerprint` (tuning one silently reuses stale caches); review_01's
instruction to calibrate was answered by config-exposing three and calibrating none.
Meanwhile small-n numbers are hardening into doctrine:

- "PaddleOCR out-recognises Apple Vision" rests on 4 hand-labelled pages plus one
  statistical tie (SROIE Δ0.006, n=12), yet is canonised as a standing memory note.
- The Segro n=1 page has become "our reading-order quality," cited as a dependency in
  rapidocr_eval_plan.md and engines/rapid.py.
- **Confidence-gated escalation — a design pillar of routing.md — ships disabled**
  (`escalate_below=0.0`) and has never been evaluated.
- The DPI cuts (ocr_det 200→150) were validated on **5 pages against VLM-generated
  pseudo-ground-truth** — no degraded scans or small print, the regimes where DPI
  bites — and are now a shipped default.
- Thai — a named motivating case — has **zero accuracy numbers** (the route is
  verified to *run*, not to be *right*; `eval_labels/thai-scan.txt` is empty).

The per-claim caveats in the docs are honest; the cross-references stopped carrying
them.

## 4. Performance

**Substantial headroom, mostly without quality cost — the biggest costs are
structural, not model-bound.**

1. **Everything is serial.** Stages in sequence, pages in a `for` loop, one job at a
   time, synchronous VLM calls. Overlapping the CPU-bound PaddleOCR track (~40%) with
   the GPU/ANE-bound VLM track (~38%), and/or page-level parallelism, is the single
   most obvious win — up to ~2× from scheduling alone, zero quality cost.
2. **Engine generation:** PP-OCRv6 tiny/small (claimed 6× on M4) first — cheaper
   experiment than RapidOCR, no port risk — then RapidOCR-on-ORT if it still earns it.
3. **The fusion DP** is GIL-bound quadratic pure Python; port the same algorithm to
   `rapidfuzz` primitives. Today it gets worse on exactly the dense pages that are
   already slow, and its overflow fallback degrades quality.
4. **The VLM script probe** (~14%) — already planned for replacement; the survey adds
   nothing better than the existing options (Vision on a crop / classifier / corpus
   config).
5. **Quality cost of past speedups:** re-check the shipped 150-DPI default on
   FUNSD/SROIE (real GT) before trusting it corpus-wide; measure the 4-bit quant delta.

**Is very-Python right? Yes.** The heavy lifting is already C++/Metal (Paddle, ONNX,
MLX); Python is the orchestration layer. The two places Python itself hurts are the
fusion DP (fix with rapidfuzz, not a rewrite) and the GIL preventing in-process
parallelism (fix with process/queue-level parallelism, which the job model almost
supports). A compiled-language rewrite would buy little and cost the ecosystem access
that is the whole point.

## 5. Flaws, omissions, oversights

Beyond reviews 01/02 (whose mechanical findings were all fixed, traceably — see §7):

- **Silent corpus-wide degradation:** `vlm_read._read` / `table_read._read` /
  `language._probe_script` catch every exception (except AirgapError) and return `""`
  — and **there is no logging anywhere in those stages** (the only logger in src is
  ocr_det.py). A misconfigured model name or dead MLX server degrades every page to
  raw det_text with zero signal. For a defensibility tool, "the reader was
  unreachable" must be loud.
- **Silent document loss (API):** two uploads named `report.pdf` with different
  content — the second overwrites the first's bytes in `in/` before the worker drains;
  job 1 sits `queued` forever, its document gone (api.py ~L109). Uploads need
  digest-/UUID-keyed names.
- **Orphaned `running` jobs:** a worker killed between `claim` and `set_status` leaves
  the job `running` forever; no lease/heartbeat/timeout (jobs.py ~L60).
- **Watcher loop dies on vanished files:** `src.stat()` / `sha256_of(src)` outside the
  try/except; a file moved between `iterdir` and use kills the whole worker
  (watcher.py ~L52–63) — the exact race the settle-gate design invites.
- **Fusion correctness edge:** a PUA-contaminated text-layer segment is superseded
  only if an OCR box overlaps it at IoU ≥ 0.5; PaddleOCR's over-segmentation routinely
  never reaches that, so contaminated garbage can survive into `best_text` in the
  final index (fusion.py ~L273). The Thai-PUA case that motivated the mechanism can
  leak through it.
- **Page-level script routing:** a Latin Bates stamp / "EXHIBIT 12" header over a Thai
  scanned body classifies the page Latin from the text layer and skips the probe
  (language.py ~L47) — wrong recogniser for the body. Routing is per-page; the docs'
  framing is per-region.
- **Confidential text in /tmp:** eval paths write rendered pages and full recovered
  text to `tempfile.mkdtemp`, never cleaned (harness.py ~L73, labels.py ~L156,
  datasets.py ~L140). Fine for benchmarks; dangerous the day someone runs `--labels`
  on a real confidential document.
- **`PATCH /config` is a no-op for processing** in the documented two-process
  deployment: it mutates the API process's Config; the separate worker never sees it
  (api.py ~L139).
- **The eval measures the wrong artifact for the headline claim:** the harness CER
  scores the *ungated* `page.vlm_reading` (harness.py ~L49), not the fused/gated
  output — the published numbers mostly benchmark the VLM, while the fusion machinery
  is scored only by the n=4 searchability sub-metric. And `insertion_rate` — the
  purpose-built hallucination proxy in metrics.py — is computed and **reported
  nowhere**.
- Smaller: `image_to_pdf` embeds photos at native pixel size = points (a 20 MP phone
  JPEG → ~420 MB pixmap per raster, blowing the cache budget; no downscale cap);
  raster cache keyed on `pdf.name` (empty for memory docs — latent collision);
  `populate_table_html` only fills literal `<td...></td>` (a `<td> </td>` yields a
  silently empty table); `vlm_read`'s refusal length-check counts only
  `source=="paddle"` chars, so it is disabled on Apple-Vision-routed pages; fused
  cluster segments replace per-detection segments in the final index (per-line
  boxes/conf survive only in mid-pipeline snapshots — at odds with the "nothing
  overwrites raw output" framing); overlay `place()` swallows all exceptions per
  fragment.

## 6. Has someone already built this?

**No — checked adversarially against the mid-2026 field.** No project ships the
package: per-region routing across deterministic + VLM engines, *reconciliation*
between them, ink-gated anti-hallucination fusion, searchable-PDF overlay with
per-segment provenance, superseded-not-discarded segments, fully offline on Apple
Silicon. Nearest misses:

- **PaddleOCR-VL** (Baidu): our geometry/semantics split — deterministic layout
  (PP-DocLayoutV3) + 0.9B VLM reader, official MLX path — but the VLM's reading is
  authoritative: no cross-check, no gate, no overlay. It *validates* the architecture.
- **MinerU**: hybrid backend adopts the triage thesis (for our stated reason —
  hallucination) but page-level, no reconciliation, no overlay. Runs on Apple Silicon
  via MLX (fragile version pinning). Highest momentum.
- **Docling** (IBM/LF AI): the strongest "why not just use X?" — engine-pluggable
  (incl. ocrmac and RapidOCR), TableFormer, a provenance-bearing document model, big
  community. But its VLM path is pipeline-*or*-VLM (granite-docling-258M, whose own
  card admits hallucination risk) with no fusion, and searchable-overlay output is an
  open feature request. **The project to watch** — if it grows overlay output and
  cross-engine reconciliation, this question gets hard.
- **Unstructured**: element-level VLM routing (Feb 2026) — same thesis — but
  commercial-platform-only, not offline-Mac-first.
- **The end-to-end doc-VLM wave** (Chandra, olmOCR-2, DeepSeek-OCR-2, GLM-OCR,
  HunyuanOCR…): consolidating the markdown-for-RAG use case, where a mangled cell is
  an inconvenience. Their own docs concede the evidentiary failure modes (blank-page
  hallucination, fabricated fields, repetition collapse). Structurally, a single
  network emitting both text and its grounding boxes can never provide "the boxes
  don't come from the thing that can hallucinate" — **the ink-gate is an architectural
  property no end-to-end model can replicate.** That is the moat, and it is real.
- **OCRmyPDF** remains the only ecosystem doing searchable-overlay-with-provenance —
  single-engine, no VLM, no routing.

The honest caveat: the moat only exists if the fusion guarantees are *measured*. Today
the differentiator is implemented but unquantified — the defensibility advantage over
"just run MinerU-hybrid" is currently an argument, not a number.

## 7. Prior-review follow-through (for calibration)

Every *mechanical* finding from reviews 01/02 was fixed, traceably (recipe
fingerprint, airgap getaddrinfo/connect_ex/AF_UNIX, watcher settle gate + move-out,
WAL + atomic upsert, digest passthrough, python-multipart, overlay_font fingerprint,
API auth/path-traversal/offload). The two dropped findings were both *evidential*:
calibrate the thresholds; measure the quant cost. The pattern is consistent — this
project fixes what a reviewer can point at in code, and defers what requires running
the eval at scale. That is the pattern to break.

## Prioritized recommendations

1. **Spend the gold data already on disk.** Full FUNSD (199) + SROIE (973) runs; fix
   the IAM transcription source (unblocks 1,539 handwriting pages — the n=1 headline
   capability); report `insertion_rate`. Converts the two biggest weaknesses at
   near-zero engineering cost.
2. **Build the one missing metric that justifies the architecture:** box-placement
   accuracy — does a searched word highlight the right line? Without it the product's
   central promise is untested.
3. **Make VLM failure loud** (logging + a per-page `read_failed` provenance flag) and
   fix the three document-loss bugs (upload collision, orphaned `running`, watcher
   loop crash).
4. **Engine refresh under the existing A/B harness:** PP-OCRv6 tiny/small first (may
   moot RapidOCR), PP-DocLayoutV3, then PaddleOCR-VL-0.9B and Vision's
   `RecognizeDocumentsRequest` as candidate tiers.
5. **Swap fusion/metrics internals to rapidfuzz/jiwer** (same algorithms, C speed,
   kills the quality-degrading DP fallback); fix or drop the word-overlay mode.
6. **Pipeline the paddle and VLM tracks** for the structural ~2×.
7. **Re-scope the airgap claim:** document the Python seal as a tripwire; ship an
   OS-level recipe (pf / sandbox-exec / network-less user) for the sealed tier.
